## 决策记录

| 时间 | 决策项 | 背景/原因 | 影响范围 |
|-----|-------|----------|---------|
| 2026-05-31 | **统一序列化方案为 kotlinx-serialization** | 项目存在两套序列化方案：Gson（app模块）和kotlinx-serialization（editor-core模块），维护成本高 | app模块、editor-core模块 |
| 2026-05-31 | **渲染层采用 Compose Canvas 方案** | 项目使用 Compose，原 ViewGroup 方案不适用 | 编辑器层、舞台层 |
| 2026-05-31 | **横竖屏方案改用 Compose WindowSizeClass** | 原 configChanges 手动处理是 View 体系方案，Compose 项目不适用 | UI层、布局切换 |
| 2026-05-31 | **历史记录暂时存内存，会话结束丢弃** | 当前编辑器功能尚未稳定，持久化成本高 | editor-core 模块 |
| 2026-05-31 | **补充国产ROM适配说明** | 小米/华为等ROM会忽略 START_STICKY，直接杀后台，需要引导用户开启自启动 | 悬浮窗层 |
| 2026-05-31 | **加入日志系统与按键记录** | 需要完整的错误追踪和操作日志 | app模块（新增 logging/KeyEventLogger.kt） |
| 2026-05-31 | **优先跑通核心链路，防止范围蔓延** | 架构设计文档虽然覆盖很全，但不要被非核心功能（如舞台交互规则）提前消耗精力 | 整个项目 |
| 2026-05-31 | **先做WebP集成验证，再开发导出器** | WebP编码方案决策拖得越久，导出器就越没法动工，需要先快速验证 | render-engine、导出器 |
| 2026-05-31 | **先View侧渲染器，再Compose侧** | View侧悬浮窗和Compose编辑器的渲染器消费端有差异，不要两头同时开工，优先做View侧 | floating-preview-base、render-engine |
| 2026-06-02 | **架构升级为多面板交互系统** | 原有小窗预览定位过窄，产品定位升级为用户自定义手机多面板交互系统 | 全项目 |
| 2026-06-02 | **悬浮窗定位改为全屏交互面板层** | 原透明画布只做视觉装饰，需要支持每个面板独立配置触摸穿透/拦截，并在全屏模式下编排 | floating-preview-base |
| 2026-06-02 | **舞台定位改为空间/交互/状态编排器** | 原舞台只管理项目叠加顺序，无法满足多面板的空间关系与交互关系编排需求 | 舞台层、floating-preview-base |
| 2026-06-02 | **面板定义为三层架构（视觉/交互/行为）** | 原项目只存动画数据，无法表达独立的触摸行为配置和事件响应逻辑 | editor-core、floating-preview-base |
| 2026-06-02 | **坐标系统一为归一化百分比（0.0~1.0）** | 存储绝对像素值无法跨设备适配，编辑器操作、渲染器入口统一做归一化→实际像素转换 | 全项目 |
| 2026-06-02 | **移除小窗模式** | 产品形态固定为全屏，小窗模式及一切相关逻辑不再需要 | floating-preview-base |
| 2026-06-02 | **行为层设置复杂度边界** | 防止行为层演变为脚本引擎，明确支持条件触发/简单序列/面板通信，不支持循环/嵌套/网络请求 | 行为层设计 |
| 2026-06-02 | **面板间事件限制载荷为简单键值对** | 防止面板间耦合过深，保持通信轻量可预期 | 事件系统设计 |
| 2026-06-02 | **交互区域与视觉层共用同一套归一化坐标系** | 确保触摸命中测试与视觉渲染位置完全对齐，渲染器和触摸分发器共用 CoordConverter | 面板设计 |
| 2026-06-02 | **明确区分两条状态同步通道** | 主进程内编辑器→舞台预览走实时StateFlow，悬浮窗运行时更新走导出后FileObserver，避免通道混淆 | 状态同步策略 |
| 2026-06-02 | **资产引用机制定型** | 面板间不共享资产，projects/{projectId}/assets/{panelId}/，AssetRef(type/localPath/fallback)，缺失静默隐藏不崩溃 | editor-core（Phase 0） |
| 2026-06-02 | **手势冲突仲裁规则定型** | ACTION_DOWN锁定目标面板，MOVE/UP持续派发不切换，锁定面板未识别手势时静默丢弃不穿透 | 舞台层（Phase 2） |
| 2026-06-02 | **PanelFactory接口定型** | StageEngine持有PanelFactory引用，createPanel(panelData): PanelView，VisualType映射到对应PanelView子类 | 舞台层（Phase 2） |
| 2026-06-02 | **空状态定义** | 无激活面板时全透明全穿透，不提示不销毁Service | floating-preview-base（Phase 1） |

## 核心链路实施计划（v2 多面板交互系统）

### Phase 0: 数据层先行（核心类型定义）
- 目标：完成所有新数据类型定义（NormalizedCoord / PanelData / StageData / PanelEvent / BehaviorAction）
- 影响模块：editor-core（新增数据包，不影响现有代码）
- 优先度高：交互区域对齐、面板间事件格式、行为层边界

### Phase 1: 悬浮窗全屏化重构
- 目标：小窗预览 → 全屏交互面板层
- 关键变更：FLAG_NOT_TOUCHABLE → 面板级触摸配置
- 前置条件：Phase 0 完成

### Phase 2: 舞台引擎 + 完整面板系统
- 目标：实现舞台三个核心职责 + 面板三层完整实现
- 交付：StageEngine / 三个编排器 / InteractionLayer / BehaviorLayer / PanelRenderer
- 前置条件：Phase 1 完成

### Phase 3: 编辑器适配 + 跨设备适配
- 目标：将新架构暴露到编辑器，实现全链路编排
- 交付：归一化编辑 / stage.json新格式 / 舞台设置界面 / 舞台预览 / 跨设备适配
- 前置条件：Phase 0, 2 完成
